# 友達旅行用 Web アプリ デザイン指示書

## 1. 目的

本ドキュメントは、友達旅行用 Web アプリに一貫したデザインルールを与えるためのデザイン指示書である。Codex で実装する際に、ページごとの見た目がばらつかないよう、配色・レイアウト・コンポーネント・雰囲気を統一することを目的とする。

このアプリは GitHub Pages 上で動作するスマホファーストの静的 Web アプリであり、旅行中にスマホで見やすく、かつ「ちょっとかっこいい」印象を持たせる。

---

## 2. デザインコンセプト

### 2.1 キーワード

- Cool
- Active
- Clean
- Sporty
- Minimal
- Outdoor

### 2.2 テーマ

**テニスやスノーボードをする仲間の、アクティブでクールな旅行アプリ**

旅行アプリとしての見やすさを保ちつつ、以下のような空気感を表現する。

- テニスウェアのようなシャープさ
- スノーボードのようなスピード感
- 雪山・青空・ナイター照明を連想させるクリーンな寒色系
- スポーティーだがゴテゴテしすぎない洗練された UI

### 2.3 デザインの方向性

- ベースは **ダークネイビー + アイスブルー + 白**
- 背景は明るすぎる真っ白ではなく、ほんの少し青みがかったクールグレー
- 情報の視認性を最優先し、装飾は控えめ
- ただしヒーロー領域や見出しでスポーティーな雰囲気を出す
- 「旅行のしおり」と「スポーツブランド風 UI」の中間を狙う

---

## 3. 想定するユーザー体験

### 3.1 利用シーン

- 旅行前に日程やホテル情報を確認する
- 旅先でスマホからスケジュールを見る
- 旅行後に経費や麻雀の精算結果を確認する
- 過去旅行を見返して思い出を振り返る

### 3.2 UX 方針

- 片手操作しやすい
- 情報にすぐたどり着ける
- 重要情報がひと目でわかる
- 金額や時刻などの「数字」が見やすい
- 旅行ごとのページに入るとテンションが少し上がる

---

## 4. ビジュアルトーン

### 4.1 全体印象

- かっこいい
- すっきり
- 爽やか
- 少し非日常感がある
- 若干スポーツブランドっぽい

### 4.2 避けたい方向性

以下のようなテイストは避ける。

- かわいすぎる
- ポップすぎる
- 観光パンフレット風に情報過多
- ビジネスアプリすぎて無機質
- スポーツチームの公式サイトのように派手すぎる
- アニメ調・イラスト過多

## 5. カラーパレット

### 5.1 主要カラー

```txt
Primary / Navy:        #10233F
Primary Dark:          #0A172B
Primary Light:         #1E3B63
Accent / Ice Blue:     #4FC3F7
Accent Dark:           #1BA6E0
Accent Mint:           #66E3D3
Background:            #F4F8FC
Surface:               #FFFFFF
Surface Alt:           #EAF1F8
Text Main:             #142033
Text Sub:              #5C6B80
Border:                #D7E0EA
Success:               #19B36B
Warning:               #F5A524
Danger:                #E34D59
Info:                  #2F80ED
```

### 5.2 色の使い分け

- **Primary / Navy**: ヘッダー、主要ボタン、見出し、下部ナビのアクティブ状態
- **Accent / Ice Blue**: リンク、強調、タグ、アイコン、グラデーションの片側
- **Accent Mint**: サブ強調、スポーティーなアクセント、選択状態の補助色
- **Background**: ページ全体背景
- **Surface**: カード背景
- **Danger / Warning / Success**: 支払い・注意・完了などの状態表示

### 5.3 グラデーション

ヒーロー領域や旅行カードのヘッダーなど、一部で以下の軽いグラデーションを使用可。

```txt
Hero Gradient:
linear-gradient(135deg, #10233F 0%, #1E3B63 55%, #4FC3F7 100%)
```

使いすぎず、アプリ全体の 10〜15% 程度に抑える。

## 6. タイポグラフィ

### 6.1 基本方針

- 日本語が読みやすいサンセリフ
- 英数字はややスポーティーに見えるものが望ましい
- 可読性優先

### 6.2 推奨フォント

```css
font-family:
  "Inter",
  "Noto Sans JP",
  "Hiragino Sans",
  "Yu Gothic UI",
  sans-serif;
```

### 6.3 文字サイズ

モバイル基準：

- Page Title: 28px / 700
- Section Title: 20px / 700
- Card Title: 18px / 700
- Body Large: 16px / 500
- Body: 14px / 400
- Caption: 12px / 400
- Numeric Highlight: 24px / 700

### 6.4 文字ルール

- 見出しは太めで短く
- 本文は 14〜16px を基本とする
- 金額、日付、時刻、スコアは数字を大きく見せる
- 説明文は長くしすぎず 2〜3 行程度で区切る

---

## 7. レイアウト原則

### 7.1 基本レイアウト

- モバイルファースト
- 最大幅は PC 時に `960px` 程度
- 画面中央寄せ
- コンテンツ左右余白はモバイルで `16px`、タブレット以上で `24px`〜`32px`

### 7.2 間隔ルール

8px グリッドを基本とする。

- 4px: 極小の隙間
- 8px: ラベルと値の間
- 12px: 小コンポーネント間
- 16px: 標準余白
- 24px: セクション内余白
- 32px: セクション間余白
- 40px: 大きな区切り

### 7.3 角丸

- 小要素: 10px
- 標準カード: 16px
- 大きいヒーローカード: 20px
- Pill / Tag / Button: 9999px または 999px

### 7.4 シャドウ

強すぎないソフトシャドウを使う。

```css
box-shadow: 0 10px 30px rgba(16, 35, 63, 0.08);
```

hover 時は少しだけ強くしてよい。

## 8. コンポーネント設計

### 8.1 App Shell

全ページ共通の外枠。

構成：

- 上部ヘッダー
- メインコンテンツ
- 下部ナビゲーション（スマホ中心）

ヘッダー方針：

- 高さは 56〜64px
- 背景は Primary / Navy
- タイトルは白
- 必要なら戻るボタンを左側に配置

### 8.2 Hero Card

トップページや旅行詳細ページ冒頭で使う目立つカード。

- グラデーション背景
- 白文字
- 旅行名を大きく表示
- 日程や場所をサブ情報として表示
- 背景に薄く斜線や斜めラインを入れて、スノボ・スポーツ感を少し出してよい

### 8.3 Standard Card

最も多く使う情報カード。

- 背景は白
- 角丸 16px
- 軽い影
- 上部に小さなアイコンやラベルを置いてもよい
- タイトル + 本文 + 補足情報 という構成を基本とする

使用例：

- 旅行一覧カード
- ホテルカード
- 経費カード
- スケジュールイベントカード
- 麻雀結果カード

### 8.4 Section Header

- 左揃え
- 太字
- 必要に応じて小さなアクセントバーを左に置く
- 補足テキストを下に入れてよい

### 8.5 Button

種類：

1. Primary Button
2. Secondary Button
3. Ghost Button
4. Icon Button

Primary Button:

- 背景: Primary / Navy
- 文字: 白
- 高さ: 44px 以上
- 角丸: 999px

Secondary Button:

- 背景: Surface Alt
- 文字: Primary / Navy
- 枠線: Border

Ghost Button:

- 背景なし
- 文字: Accent / Ice Blue

### 8.6 Tag / Chip

カテゴリや状態表示に使う。

例：

- 計画中
- 実施済み
- ホテル
- 食費
- 交通費
- 精算済み

デザイン：

- 丸みの強い Pill 形状
- 背景薄色 + 文字濃色
- 色分けはわかりやすくする

### 8.7 Bottom Navigation

スマホで主要ページを移動しやすくするための下部ナビ。

項目：

- 概要
- スケジュール
- ホテル
- 経費
- 麻雀

デザイン：

- 背景: 白 + 上罫線
- アクティブ色: Primary / Navy または Accent / Ice Blue
- アイコン + ラベル構成
- 高さは 64〜72px 程度

### 8.8 List Item / Row

- 左にラベル、右に値
- 重要な値は右寄せ・太字
- 区切り線は薄く
- スマホ時は詰め込みすぎず、2 行構成も許容

### 8.9 Timeline Item

スケジュール表示用。

- 左に縦線
- 時刻を目立たせる
- 各イベントはタイムライン＋カード
- 時刻、イベント名、場所、メモを表示する

### 8.10 Settlement Summary Card

割り勘や麻雀精算の結果用の目立つカード。

- 背景は白
- 収入系は Success、支払い系は Danger をアクセントに使う
- 金額は大きく、数字を見やすくする
- 「誰が誰にいくら払うか」が一目でわかることを最優先する

## 9. アイコン方針

### 9.1 アイコンのトーン

- 線画系またはシンプルなソリッド系
- 太さは統一する
- ゴテゴテしたものは避ける

### 9.2 使用候補

- Home
- Calendar / Schedule
- Bed / Hotel
- Wallet / Receipt / Coins
- Trophy / Mahjong / Gamepad
- MapPin
- Car
- Train / Plane
- Users

### 9.3 実装

必要なら `lucide-react` のようなミニマルなアイコンセットを使う。

## 10. 画像・装飾の方向性

### 10.1 背景装飾

アプリ全体で写真を多用する必要はないが、以下のようなスポーティーな抽象表現は使用可。

- 斜めライン
- 薄いグラデーション
- 山や雪面を連想させる幾何学的な形
- テニスコートのラインを抽象化したパターン

### 10.2 実写表現

実写写真を必須にはしない。もし使う場合も、背景ヒーローに薄く使う程度にとどめる。

### 10.3 装飾ルール

- 1 画面に装飾を入れすぎない
- 文字の可読性を損なわない
- 装飾より情報が主役

## 11. ページごとのデザイン指針

### 11.1 トップページ

目的：旅行一覧を見せ、今年の旅行にすぐ入れるようにする。

構成：

1. Hero
   - アプリ名
   - 一言キャッチコピー
   - 最新旅行への CTA
2. 最新旅行カード
3. 過去旅行一覧

印象：

- 最もスポーティーな雰囲気を出してよいページ
- ただし派手すぎず、落ち着いたクールさを優先する

キャッチコピー例：

- 「Ride the Trip.」
- 「Plan. Move. Settle. Repeat.」
- 「仲間との旅を、クールにまとめる。」

### 11.2 旅行詳細ページ

目的：各機能へのハブ。

構成：

- 旅行名ヒーローカード
- 日程・行き先・参加者概要
- 機能導線カード 4〜5 個

UI：

- 機能カードは押しやすく大きめ
- それぞれにアイコンをつける
- スマホでは 1 列、広い画面では 2 列でも可

### 11.3 スケジュールページ

- 日ごとにセクションを分ける
- 日付の見出しを大きく出す
- 各イベントはタイムライン＋カード
- 移動や集合などの重要イベントは少し強調してよい

### 11.4 ホテルページ

- ホテル名を大きく表示
- チェックイン / チェックアウトを目立たせる
- 住所、Map リンク、公式サイトはカードまたはリストで整理
- 部屋割りはメンバー一覧を見やすく整理

### 11.5 経費・割り勘ページ

- サマリーカードで総額・立替総額・精算件数などを出してもよい
- 経費一覧はカテゴリタグ付きカード
- 最終精算結果は最も目立つ形で表示
- 「支払う」「受け取る」がひと目で判別できる色分けにする
- 金額の視認性が最重要

### 11.6 麻雀ページ

- スコアカードと最終精算カードを分ける
- 順位は視覚的に区別する
- 1位のみ少し強調してもよい
- ギャンブルサイト風にしない
- 旅行アプリの一機能として統一感を保つ

---

## 12. 状態表現

### 12.1 ステータス

- 計画中: Ice Blue 系
- 実施済み: Navy 系
- 注意事項: Warning
- 支払い: Danger
- 受け取り: Success

### 12.2 空状態

データがない場合も、冷たい印象にならないようシンプルな案内を出す。

例：

- 「まだスケジュールが登録されていません」
- 「経費データはあとで追加できます」
- 「麻雀結果がまだありません」

---

## 13. インタラクション方針

### 13.1 アニメーション

- 基本は控えめ
- hover / tap 時に少し浮く程度
- ページ遷移演出は不要または最小限
- スクロールアニメーションを多用しない

### 13.2 体感

- 軽い
- すぐ押せる
- 迷わない

---

## 14. アクセシビリティ指針

- 背景と文字のコントラストを十分に確保する
- 色だけで状態を区別せず、ラベルやアイコンも併用する
- ボタンのタップ領域は十分に確保する
- 日本語の可読性を優先し、文字を小さくしすぎない

## 16. CSS 実装方針

- コンポーネントごとにスタイルを再利用できるようにする
- 配色や余白はハードコードしすぎず、トークン化する
- デザインの統一感を保つため、都度別デザインを作らない
- CSS Modules、Tailwind CSS、styled-components のいずれでもよい
- Tailwind を使う場合も colors / spacing / radius を theme 拡張で寄せる

---

## 17. Codex への実装指示

Codex に実装させる際は、以下の点を必ず守る。

1. 本 `design.md` のトーンに沿って全ページを実装すること
2. スマホ表示を最優先し、レスポンシブ対応すること
3. コンポーネントの使い回しを前提にすること
4. カラー、余白、角丸、影を統一すること
5. 旅行詳細、経費、麻雀などページごとに別世界のようなデザインにしないこと
6. 情報の見やすさを最優先し、装飾過多にしないこと
7. Hero やカードで「テニス・スノボのクールさ」を表現してよいが、機能性を損なわないこと
8. 数字・時刻・金額の視認性を高くすること

---

## 18. 一言まとめ

**「テニスやスノーボードが好きな仲間向けの、スマホで見やすい、クールでクリーンな旅行アプリ」**

スポーティーで少し高揚感がありつつ、旅行中・旅行後に本当に使いやすい情報整理アプリとして仕上げる。
